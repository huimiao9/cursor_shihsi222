# PRD 写作助手

基于字节跳动/SOLO 团队的 PRD 文档规范，帮助用户快速生成结构清晰、内容完整的产品需求文档。

## 核心能力

1. **生成 PRD 结构框架** - 根据功能类型提供对应的文档模板
2. **需求背景撰写** - 帮助梳理 What/Who/Why、现状问题、目标
3. **需求描述生成** - 功能模块表格化、场景化描述
4. **埋点需求整理** - 数据追踪点梳理
5. **竞品调研框架** - 对比分析表格

[这个功能是为谁构建的]

**现状 & 问题**
- [当前存在的问题]

## 指令

### `+create` - 创建新 PRD

根据用户提供的功能描述，生成完整的 PRD 框架。

**用法：**
```
prd-writer +create "功能描述" --type=[feature|improvement|integration]
```

**示例：**
```
prd-writer +create "在DiffView中增加文件树功能，支持代码审查" --type=feature
```

**输出：**
- 完整的 PRD 文档结构
- 中英日三语占位符
- 表格模板
- 埋点框架

### `+background` - 生成需求背景

帮助撰写「需求背景」章节。

**用法：**
```
prd-writer +background "功能描述"
```

**输出：**
- What/Who/Why 三段式描述
- 现状问题分析
- Primary Goals

### `+module` - 生成功能模块表格

生成需求描述中的功能模块表格。

**用法：**
```
prd-writer +module "模块1:描述1,模块2:描述2"
```

**输出：**
- 标准三列表格（模块/需求说明/原型参考）
- 中英日三语文案占位

### `+tracking` - 生成埋点需求表格

生成数据埋点追踪表格。

**用法：**
```
prd-writer +tracking "页面1,页面2,页面3"
```

**输出：**
- 五列表格（序号/页面/图片/埋点/意义）
- 常用埋点事件建议

### `+competitor` - 生成竞品调研框架

生成竞品对比分析表格。

**用法：**
```
prd-writer +competitor "竞品1,竞品2,竞品3"
```

**输出：**
- 竞品对比表格
- 分析维度建议

## 写作规范

### 文案规范

1. **中英日三语**：所有用户-facing文案需提供中文、英文、日文
2. **格式**：`中文/English/日本語`
3. **占位符示例**：
   - 「显示文件树/Show File Tree/ファイルツリーを表示」
   - 「保留所有变更/Keep all changes/すべての変更を保持」

### 表格规范

1. **功能模块表**：模块 | 需求说明 | 原型参考
2. **埋点需求表**：序号 | 页面/模块 | 图片参考 | 埋点需求 | 意义
3. **竞品调研表**：竞品 | 用户故事 | 功能对比 | 借鉴点

### 需求描述规范

1. **分场景描述**：明确区分不同使用场景
2. **状态说明**：列出所有可能的交互状态
3. **入口说明**：清晰描述功能入口位置
4. **交互流程**：步骤化描述用户操作路径

### 埋点规范

1. **事件命名**：动词+名词，如 `button_click`, `page_view`
2. **观测指标**：点击率 = 点击/曝光
3. **意义说明**：说明数据用途和决策依据

## 示例

### 完整 PRD 示例

```markdown
# PRD | DiffView 改造

## 1. 需求背景

### What are we building?
DiffView 工具新增文件树功能，支持在代码审查场景下更高效地浏览和处理多文件变更。

### Who's it for?
有 code review 需求的专业开发者用户

### Why build it?
**现状 & 问题**
- 当前 editor 中只能看到单个文件的 diff
- 用户操作路径长，效率低
- 代码审查场景由不同工具承载，理解成本高

**优化方案**
- DiffView 统一承载代码审查场景
- 增加文件树，支持多文件浏览
- 支持不同粒度的 keep/undo 操作

### Primary Goal(s)
- 缩短用户在「处理待审查的变更」场景下的操作路径
- 改善「查看历史变更快照」场景下的用户体验

## 2. 需求描述

**需求范围：** SOLO + IDE；国内 + 海外；内场 + 外场；toC + toB
**设计稿：** https://www.figma.com/...

| 模块 | 需求说明 | 原型参考 |
|------|----------|----------|
| 岛台入口 | 岛台增加「Review/审查/レビュー」按钮，点击打开 DiffView | [截图] |
| 文件列表 | 左侧显示文件树，支持展开/隐藏；点击文件右侧定位 | [截图] |
| Diff 展示 | 右侧按文件显示 diff，支持 keep/undo/编辑 | [截图] |

## 3. 埋点需求

| 序号 | 页面/模块 | 图片参考 | 埋点需求 | 意义 |
|------|-----------|----------|----------|------|
| 1 | 岛台 | [截图] | Keep All/Undo All/Review 按钮点击 | 功能渗透 |
| 2 | DiffView | [截图] | 右上角 Keep All/Undo All 点击 | 正负反馈分析 |

## 4. 竞品调研

| 竞品 | 用户故事 & 交互流程 | 功能对比 | 借鉴点 |
|------|---------------------|----------|--------|
| GitHub | 左侧文件树 + 右侧 diff，支持多文件 review | ✅ 文件树 ✅ 多文件浏览 | 文件树交互 |
| GitLab | MR review 流程，支持 suggestion | ✅ 建议修改 | 代码建议功能 |
```

## 使用建议

1. **先框架后细节**：先用 `+create` 生成整体框架，再逐步填充内容
2. **多语言同步**：中文定稿后再翻译英文和日文
3. **图文结合**：每个功能点尽量配上原型图或截图
4. **数据驱动**：埋点需求与功能目标对应
5. **竞品对标**：调研竞品时关注差异化价值
